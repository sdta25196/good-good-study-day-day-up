# 吴恩达Agent

## 一：拆分任务步骤流程

1. 拆分为足够细分的步骤（任务要尽可能的拆解为最小的可执行粒度）

2. 保证这件事不是用现有软件或者工具可以实现的，也就是这个任务**能用AI并且只能用AI**。

## 二：Agent设计模式

反思（Reflection）、工具使用（tool-use）、规划(planning)、多智能体协作（multi-agentic workflows）

### Reflection反思

反思通常的实现逻辑为：把执行结果给大模型让大模型进行反思哪里不好，然后提供一个更好的执行结果。

反思改进任务成果；通常会带来一些提升。配合 few-shot 也可以提升任务成果。

**注意事项**

1. 评估反思确实带来了提升

要先验证确实由于反思带来的提升。有些任务不用反思就可以做到很好的情况下，加入反思只会带来额外的负担。

2. 额外的时间损耗
  
反思带来额外的时间消耗。系统会因此变慢 

3. 利用外部反馈

有些任务不是纯粹的反思能够解决的。这时候最好配合外部的反馈来进行反思；

例如：按照字数要求写作，大概率AI写的字数是不满足的，这时可以考外部的字数检查器来检查结果，然后作为信息给到反思提示词，大模型的反思会更加有效。

4. 用评分逻辑代替选项

有的任务反思是需要对比判断，此时要注意：大模型在面对选项时，对位置是存在偏见的，选择第一个选项的频率比选择第二个更高。

所以尽量让大模型**按照明确的标准**进行评分来代替选择，同时评分标准要尽量**控制在0或者1**，**不要用1-5**这样的跨度评分；

例如：

```bash
根据该质量评分标准评估所附图片。每个项目应根据是否符合（true/false）获得 1 分或 0 分。将每个项目的得分以 JSON 对象形式返回。
1. 是否有清晰标题
2. 是否包含坐标轴标签
3. 图表类型是否合适
4. 坐标轴是否使用合适的数值范围
```

### tool-use使用工具

**1. 使用function call**

让大模型可以调用某个函数来执行操作、收集信息等行为，调用大模型的function call能力。

大模型本身不调用工具，大模型只是要求你调用工具。

每个工具一个清晰明确的职责（单一路由，单一效果）。

> 调用方式、逻辑等参考【function call】即可

**2. 自动编写与执行代码**

Agent中的工具除了调用function call之外，还有一个逻辑是AI自动根据情况编写代码然后执行

例如：科学计算器的实现，我们没有必要把所有的科学计算器上的功能按键都实现。

AI编写的代码，最好使用**沙箱环境**进行执行，例如`nodejs`使用的`vm2`、`isolated-vm`

**3. 使用MCP**

调用其他的MCP服务器，以此来获得工具的使用。

> MCP的本质还是function call

## 三、eval评估

### 评估

假设你已经构建了一个智能体工作流程，但它还没有达到你预期的效果，这时就可以构建一个评估系统来协作进行效果提升。

- 评估系统从最简单的测评集开始构建，最早的评估系统可以是简陋的，但是要快速搭建完成。
- 评估系统要记录整个流程中每一个节点的输出信息
- 优先解决对Agent影响最大的问题节点。放弃我们无法解决的问题节点。
- 评估系统也可以针对组件进行评估，小规模评估和解决问题；快速、成本低、噪音少；

评估的是节点执行结果。确定性结果和不确定性结果分成两种评估方式

- 确定性的直接按照标准准备测试集就行

- 不确定性的就需要准备打分提示词进行评估。

### 问题节点解决方案：

- 非LLM节点：调整参数（网络搜索引擎调整返回数量、日期范围）、更换组件（百度搜索换必应搜索或者duckduckgo）
- LLM节点优化：
  - 提示词优化：优化指令、添加fet-show
  - 切换模型
  - 大任务拆任务拆分
  - 微调

**质量优化**：优化系统输出质量。
**延迟优化**：提高工作流程的响应速度。对工作流程的每个步骤进行计时，识别耗时最长的环节。
  - 对可并行的步骤（如多个网页搜索）实施并行执行以缩短总时间。
  - 对耗时过长的LLM步骤，可尝试更换为更小、更快的模型（在满足性能要求的前提下）
  - 优化输出响应，使用最小输出来节约输出时间
  - 寻找响应速度更快的LLM服务提供商（比如deepseek官方就不如阿里、火山的更快）。
**成本优化**：降低运行开销，尤其是在生产环境中。计算工作流程中每一步的成本。
  - 常见成本构成：LLM按输入/输出token数计费、各类API按调用次数计费、计算资源按使用量计费等。
  - 替换更便宜的服务商。
  - 优化输入、输出的tokens数量

## 四、规划和智能体

规划是一种设计模式，允许Agent在处理复杂任务时，自行决定执行步骤的顺序，而不是依赖开发者预先编写的固定流程。

Agent自主制定计划，然后分步执行，最后生成答案；

优点是：**无需为每个可能的复杂查询预先硬编码执行顺序，智能体可动态适配，处理范围更广的任务。**

缺点是：**无法完全预支智能体将生成什么计划**, 需要我们在设计Agent的时候做一些保障行为正确的矫正逻辑；

> Agent中间流程的输出尽可能使用JSON、XML、代码；输出代码的话，要确保在沙盒中运行

### ReAct

ReAct 的核心是一个不断重复的 “思考-行动-观察”（Thought-Action-Observation）循环，就像人类解决问题一样：
- Thought（思考）：模型先分析当前情况，制定下一步计划。例如：“用户问股价，我需要先查一下苹果公司的股票代码。”
- Action（行动）：根据思考结果，调用外部工具。例如：执行 search("Apple stock symbol")。
- Observation（观察）：获取工具返回的真实结果。例如：返回 AAPL。

### 多智能体（multi-agentic workflows）

多智能体系统的核心价值在于关注点分离, 将复杂任务分解给不同“专业岗位”，比让一个LLM反复切换角色更可靠。

每个智能体通过角色提示（Role Prompting）​ 构建。

两种核心协作模式:

- 线性工作流：简单、可控的模式；像流水线一样，前一个智能体的输出是后一个智能体的输入。

  例如：研究员（产出报告） → 设计师（产出图片） → 撰稿人（产出终稿）。
- 管理者-执行者模式：高级的“规划”模式，引入了管理者智能体。

  例如：一个市场经理（Manager Agent）​ 负责制定计划，并调用下属智能体（研究员、设计师等）来执行具体步骤。

通信模式：
- 线性通信：流水线一样，信息从一个智能体传递给下一个智能体，最后一个智能体负责输出
- 层级通信：中心管理者智能体，分派给下级智能体任务，并最终整合输出
- 多层级通信：多层级的通信，每个层级都有中心管理者，最高级管理者负责输出
- 全员讨论通信：会议室模式，所有智能体都参与讨论共同推进任务，最终结果由某个智能体输出；

目前应用较多的是前两种，后两种应用较少。


# 构建Agent的方法论

落地技术选择方案

**不是AI需求的时候 用传统工程逻辑解决；**
**初级AI需求，简单的推理、分析、生成, 用LLM**
**需要按照流程进行执行，需要整个任务流程都是可预测的，用工作流**
**没有固定流程可用，全靠AI自主判断执行; 工作流解决不了,用Agent**
**需要多智能体协作的时候，就升级多智能体；在此之前可以考虑Agent+skill**

## 论述如下

随着业务复杂度的上升，落地方案的演化方向为：`传统工程逻辑` => `大语言模型(LLM)` => `工作流(workflow)` => `智能体(Agent)` => `多智能体协作(multi-Agent)`。

在完成AI需求的时候，我们选择落地方案要尽可能选择简单的解决方案, 主打一个遵循奥卡姆剃刀原理，如无必要勿增实体。

当需要更高复杂性时，工作流程为明确定义的任务提供可预测性和一致性，而在大规模需要灵活性和模型驱动决策时，代理是更好的选择。

然而，对于许多应用来说，仅仅通过检索和上下文示例优化单个 LLM 调用通常就足够了。


**传统工程逻辑 和 workflow 的区别**

最大的区别在于：从自动化的流程到自动化的认知，AI在流程中执行了思考的工作（语义理解、推理、下达指令等），来达到工具流具备自动化认知的程度。

| 维度       | 传统工程逻辑           | Workflow                             |
| :--------- | :--------------------- | :----------------------------------- |
| 核心驱动力 | 规则                   | 意图                                 |
| 工作模式   | 执行                   | 思考 + 执行                          |
| 输入/输出  | 结构化输入，确定性输出 | 非结构化/模糊输入，生成式/推理式输出 |
| 流程状态   | 线性、刚性、预设路径   | 动态、柔性、自主规划路径             |
| 关键节点   | 条件判断               | 认知判断                             |

**Workflow 和 Agent 的区别**

最大的区别在于：流程是否完全由AI主导。

workflow是我们写好了程序，控制好整个流程，定义了任务执行路径，并在每个步骤中的操作方式设定了处理边界，AI可以在我们设定的边界内进行一些动态的行为。

工作流中的每个步骤都可以利用智能体的推理和工具使用能力，但整体流程遵循既定路径，也就是说**整个任务的流程是可预测的**

Agent是在整个任务执行过程中，使用哪些工具、执行任务的顺序以及何时停止进行反馈，全都由AI自主进行。

这就是AI的主导性


| 维度           | Workflow               | 智能体 (Agent)                                  |
| :------------- | :--------------------- | :---------------------------------------------- |
| **核心驱动力** | 流程编排               | 目标驱动                                        |
| **工作模式**   | 任务链                 | 思考-行动-观察循环 (Thought-Action-Observation) |
| **输入/输出**  | 任务级输入，步骤级输出 | 目标级输入，结果级输出                          |
| **流程状态**   | 有向图 依赖外部调度    | 自主状态机，具备记忆与反思                      |
| **关键节点**   | 工具调用               | 自主规划 与 工具使用                            |

**Agent 和 Multi-Agent 的区别**

最大的区别在于：多智能体拥有协作能力。

| 维度           | 智能体 (Agent)             | 多智能体协作 (Multi-Agent)  |
| :------------- | :------------------------- | :-------------------------- |
| **核心驱动力** | 个体目标                   | 群体目标 与 社会契约        |
| **工作模式**   | 单点突破                   | 分工协作 与 竞争/协商       |
| **输入/输出**  | 单一视角输入，个体决策输出 | 多源异构输入，共识/竞合输出 |
| **流程状态**   | 内部循环                   | 群体通信                    |
| **关键节点**   | 自我反思                   | 角色分配                    |


## 业务是否有容错率

- 高控制需求，比如金融交易、监管合规、安全关键操作，你需要向审计人员解释系统为什么做出某个决策。一个贷款审批系统，如果说不清楚为什么拒绝了某个申请，这个系统就不能上线。这种情况采用单体 Agent 或串行工作流，要的是可预测、可追溯的行为。
  
- 中等控制需求，比如客服、内容创作、数据分析，可以引入层级多智能体，主管 Agent 负责执行业务规则，专属 Agent 处理复杂度。

- 低控制需求，比如研究、头脑风暴、复杂分析，多智能体的不可预测性反而是优势，可以放开让它自由探索。


## 业务复杂度是否需要Agent

有标准的流程图且流程固定，就用工作流； 问题复杂到没有可工程化的标准流程后再考虑多智能体

- 单一领域的问题，比如回答产品问题、处理退货、生成报告，单体 Agent 就能高效处理，不要过度工程化。

- 多领域但流程可预测的问题，比如员工入职、合规审查、标准分析任务，你能画出完整的流程图，用串行或并行工作流就够了。

- 复杂的开放式问题，比如战略分析、研究项目、系统故障排查，连步骤都无法提前定义，才需要多智能体。


## 资源上的硬性要求是否足够

上线节奏、时间点、tokens开销等资源限制，同样决定我们采取怎样的Agent架构模式。

- tokens消耗的硬性费用是否支持：Agent消耗的tokens是workflow的5-10倍，多智能体消耗的 Token 大约是单体的 10 到 15 倍。

- 开发周期是否支持：workflow 但周内可能完成，单体 Agent 通常要几周时间，多智能体根据复杂度可能需要1-N个月才能做完。
  

## 是否需要多个领域的协作能力

- 多 Agent 解决多领域问题要协作运行的问题；
  
- 单 Agent + Skills；由skill来提供专业能力，同时避免引入多Agent而为系统带来复杂度。

文档分析就是典型例子：

初期：一个可以调用分析 Skill 的单体 Agent 就够了。
  
当需要提取关键主题、进行情感分析和事实验证协作分析时，我们就需要把系统演进成多智能体。

所以我们在开发单 Agent 时，要有业务发展的前瞻性，可选择性的预留好后续扩展成多智能体的接口。
